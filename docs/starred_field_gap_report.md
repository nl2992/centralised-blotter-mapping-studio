# Starred-Field Gap Report

Evidence-based: generated by running `cd tests && node run_starred_field_matrix.js`
against `ocr_work/test_starred_field_matrix.xlsx` (a NEW fixture with realistic,
COMPLETE rows for every asset family -- built to isolate "does the app have a code
path to fill this field at all" from "was this particular source row sparse")
and reading its actual `window.__BOARD_STAR_AUDIT()` output. Every value/className
pair quoted below was observed by the harness, not guessed. Result on the run that
produced this report: **130/130 matrix cells accounted for -- 126 PASS (real
coverage), 4 documented GAP (confirmed, not failures), 0 unexpected FAIL**
(default app settings; see `docs/asset_class_input_spec.md` for the full per-field
research and `tests/run_starred_field_matrix.js` for the expectation matrix and raw
per-cell detail lines). The harness treats a cell three ways: `PASS` (populated as
authored), `GAP` (authored as a known, code-level gap, and confirmed unpopulated/
placeholder-only), `FAIL` (anything that should have populated but didn't, or a
documented gap that unexpectedly started resolving). The 4 GAP cells below are
exactly the two fields x two asset classes identified in the code-reading pass.

## Asset classes that CANNOT reach full coverage even with complete input

### Structured Credit and Private Credit (both from `parseStructuredCredit2025`, sheet "Structured Credit 2025")

| Field | Blocked? | Observed value/className | Root cause |
|---|---|---|---|
| `*Trade Date` | **Yes -- hard gap** | `""` / `UNRESOLVED` | `parseStructuredCredit2025()` never reads any date column -- `row.tradeDate` is hardcoded `null` in the parser (no `rowValue()` call for it exists at all). The `Structured Credit 2025` sheet itself has no trade-date column either. `mapOne()`'s `*Trade Date` assignment (`setRequiredText("*Trade Date", formatDateOutput(row.tradeDate), "SOURCE_BACKED", "Source trade date")`) passes no placeholder key, so a blank value is an unconditional `UNRESOLVED`/fail. **This single field is why both asset classes can never exceed `quality: FAIL`** -- confirmed both in this matrix run and in the pre-existing `tests/expected.js` TC05/TC06 assertions (`quality: "FAIL"`, `tradeDate: ""`), which this report corroborates rather than duplicates. |
| `*Treats Acronym` | **Yes -- placeholder-only gap** | `"PENDING_TREATS_ACRONYM"` / `PLACEHOLDER` | `getBuiltInTreatsAcronym()` and `getPbRouting()` both key off `row.client`/`row.book`/`row.issuer`/`row.issuerRaw` and a `raw` lookup on `FINAL CUSTOMER`/`Sales Client`/`Customer`/`Client`/`Issuer`/`Book`. `parseStructuredCredit2025()` hardcodes `client`, `book`, `saleTeam`, `issuer`, `issuerRaw` all to `""`/absent -- and no such columns exist on the source sheet in the first place (its only columns are `Product`, `Category`, `Region`, `Notional (USD)`, `GNBV (USD)`). With no reference CSV loaded (none is required for full coverage on the other 8 classes), no built-in client match or PB-site routing match is possible, so the field falls through to the working-mode `PLACEHOLDER` under default settings (`runMode=working`, `complianceMode=pragmatic`, `allowTreatsPlaceholder=true`). Under `Controlled`/`Strict` settings (placeholder disallowed) this becomes `UNRESOLVED` instead -- either way, **it is never real coverage.** |

No other starred field is blocked for these two classes: `*Primary CCY` (hardcoded
constant `"USD"` in the parser -- populates, but see the traceability note below),
`*$ Volume`, `*$ VA/GNBV` (from `Notional (USD)`/`GNBV (USD)`), `*$ PC` (VA-proxy
policy, no CSV needed), `*Trade ID` (deterministic hash always available),
`*Revenue CCY` (constant), `*Tier 1/2/3 Product Type` (`TIER_DEFAULTS` /
`BUILT_IN_PRODUCT_TAXONOMY`), `*Salesperson (Coverage)`, `*Legal Entity` (constant
defaults) all resolved as real, non-placeholder values in this run.

### Traceability nuance (not a coverage gap, but worth flagging)

`*Primary CCY` for Structured Credit/Private Credit is recorded with className
`SOURCE_BACKED` in the audit even though it is not actually read from any source
column -- `parseStructuredCredit2025()` sets `primaryCcy: "USD"` as a hardcoded
literal (~line 2086), and `mapOne()`'s generic `setRequiredText()` call has no way
to distinguish "value came from a real source cell" from "value came from a
parser-level constant that happens to look like source data." The field is
correctly populated (not a gap), but auditors relying on the className to judge
provenance should know this one class's `*Primary CCY` is a policy constant in
parser's clothing, not a genuine source read. No other starred field across any
asset class was found to have this same mislabeling in this research pass.

## Not found: any field blocked purely by "needs a reference CSV with no built-in fallback"

Every one of the 4 reference-CSV-backed fields (`*Salesperson (Coverage)`,
`*Legal Entity`, `*Treats Acronym`, and the `*Tier 1/2/3 Product Type`
product-reference override) has a built-in, code-level fallback that does NOT
require any CSV to be loaded (`mark lok leung` / `HBAP` / `BUILT_IN_TREATS_BY_CLIENT`
+ PB routing / `TIER_DEFAULTS` + `BUILT_IN_PRODUCT_TAXONOMY`). The one case where a
field still cannot resolve (`*Treats Acronym` for Structured Credit/Private
Credit) fails not because a CSV is missing, but because the parser never captures
the input signals (`client`/`book`/`saleTeam`) that even the built-in fallback
logic needs -- a fixture/reference-data problem would not fix it; only a parser
code change (reading `FINAL CUSTOMER`/`Book`/`SALETEAM`-equivalent columns, if
added to the sheet) would.
